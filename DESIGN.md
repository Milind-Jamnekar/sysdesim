## Problem Statement

Learning distributed systems through text (GitHub's System Design Primer, books, articles) produces declarative knowledge — you can describe how caching works but can't feel what happens when cache miss rate hits 80%. The gap between "I read about this" and "I understand this" is the gap between description and simulation.

This tool closes that gap by making failure the primary frame: every concept is learned by watching it break, then fixing it.

## What Makes This Cool

Failure injection as the *default question*, not an advanced feature. Every other tool in this space (paperdraw.dev, systemdesignsimulator.org, SyDe) treats failure injection as a toggle. This tool opens with "what breaks first?" as the starting point. You kill a component, watch metrics cascade in real time, and develop intuition for the *sequence* of failures — not just the fact that they happen.

Secondary "whoa" (from second opinion): a time scrubber that lets you replay the failure sequence and scrub through it frame by frame. Deferred to v3 — the SimEngine in v1 does NOT record snapshots; replay is a clean addition once the sim is stable.

## Constraints

- Browser-only. No backend, no server, no auth. Runs from a file if needed.
- Building for self-learning first. Polish is secondary to correctness of the simulation.
- One well-instrumented scenario beats five shallow ones. Go deep before wide.
- Weekend-shippable v1.

## Premises

1. Visual simulation + live metrics (queue depth, p99 latency, error rate, throughput) together teach distributed systems better than reading alone. Not just boxes and arrows — real numbers responding to real state changes.
2. Failure injection is the primary learning frame. "What breaks first when I kill this component?" is the default question, not an advanced mode.
3. Building for yourself first. Shareable later — the tool only needs to be good enough to teach one person deeply.
4. Start with one scenario you can break and understand completely before adding more.

## Cross-Model Perspective

**Claude subagent cold read:**

Key insight: "the moment a concept you've read about 10 times suddenly clicks because you watched it happen" — this is a specific learning failure repeated. Already read the content. The gap is between declarative knowledge ("caching reduces latency") and embodied knowledge ("oh, when cache miss rate hits 80%, the DB gets 5x traffic AND p99 spikes AND the queue backs up SIMULTANEOUSLY — they compound each other"). Scope down, go deep, not wide.

Coolest version not yet considered: A **time scrubber** — after running a failure scenario, a timeline slider appears. Scrub backward and forward through the failure. Every metric is recorded per tick. Pause at the exact moment a circuit breaker trips. Replay with one variable changed and watch two timelines diverge side by side. Nobody builds replay into simulators (all live-only). The scrubber makes the failure *sequence* navigable, not just watchable.

50% solution: D3.js handles graph layout, force simulation, and real-time metric charts. The remaining 50% is a hand-rolled discrete-event simulation loop in TypeScript — each node is a state machine (healthy → degraded → failed → recovering), each tick propagates load using simple formulas: `latency = base_latency * (1 + queue_depth / capacity)`.

Prototype suggestion: Vite + TypeScript + D3.js. No React, no backend. One hardcoded scenario, one failure button, working in a weekend.

## Approaches Considered

### Approach A: One hardcoded 5-node scenario (chosen)
- **Stack:** Vite + TypeScript + D3.js
- **Scenario:** Load Balancer → App Server A → App Server B → Cache → Database
- **Mechanics:** Tick loop at 10 ticks/second. Each node has throughput (req/s), p99 latency (ms), error rate (%). Baseline: 1000 req/s, everything green.
- **Failure:** "Kill Cache" button flips cache state to failed. Miss traffic routes to DB. DB throughput spikes. Latency climbs. Error rate follows. App servers start returning 503s if DB enters degraded state.
- **Recovery:** "Recover Cache" drains metrics back to baseline over 30 seconds.
- **Effort:** 1-2 days

### Approach B: Flexible canvas with drag-and-drop topology builder
- Build any architecture, save/load scenarios
- Requires 2-3 weeks of infrastructure before the first interesting failure
- Right for v2, not v1

### Approach C: Time scrubber — record-and-replay
- Record every tick, scrub through failure sequence, compare two timelines
- The most novel idea in the space — nobody has built this
- Requires a stable simulation engine first; right for v3

## Recommended Approach

**Approach A** — one hardcoded scenario, shipped this weekend.

Rationale: matches the stated learning goal (understand one failure deeply), avoids 3 weeks of infrastructure before learning anything, and the simulation engine built here becomes the foundation for B and C. Go deep before wide.

Architecture notes:

**Separation of concerns (critical):** D3 is render-only. `SimEngine` owns the tick loop and all state. D3 reads from `SimEngine` output via a callback, never writes to it.

**Topology (hardcoded v1):**
```
[Load Balancer] → [App Server A] ↘
                → [App Server B] → [Cache] → [Database]
```
App servers are **parallel** — LB splits load 50/50. Both route reads to Cache; on cache miss, they go to DB directly.

**SimEngine interface:**
```ts
class SimEngine {
  constructor(onTick: (snapshot: Snapshot) => void)
  start(): void      // begins setInterval at 100ms
  stop(): void
  setNodeState(nodeId: string, state: NodeState): void
}

type NodeState = 'healthy' | 'degraded' | 'failed' | 'recovering'

type Snapshot = {
  tick: number
  nodes: { id: string; state: NodeState; throughput: number; p99Latency: number; errorRate: number; queueDepth: number }[]
}
```
`SimEngine` calls `onTick(snapshot)` at end of each tick. D3Renderer registers itself in its constructor.

**D3 usage:** Fixed SVG coordinates only — do NOT use D3 force simulation (adds nondeterminism; wrong for a hardcoded graph). Use `d3.select` + `d3.line` for edges, circles for nodes, `d3.scaleLinear` + `d3.axisBottom` for charts.

**Interaction states:**

| Feature | Initial | Running | After Kill | During Recovery | After Recover |
|---------|---------|---------|------------|-----------------|---------------|
| Graph nodes | All green circles, `0%` queue | Colors reflect state; queue % updates each tick | Cache → red; DB transitions amber → red | Cache → blue; DB draining | All nodes → green |
| Charts | Empty axes; lines begin at tick 1 and grow left→right | 60-second sliding window of all 4 nodes | Chart lines spike (DB throughput ↑, p99 ↑, error rate ↑) | Lines return toward baseline | All lines flat at baseline |
| Kill Cache btn | Enabled | Enabled (cache healthy) | **Disabled** (cache failed/recovering) | **Disabled** | Enabled |
| Recover Cache btn | **Disabled** | **Disabled** (cache healthy) | Enabled (cache failed) | Enabled (cache recovering) | **Disabled** |

Button disable rule: Kill Cache disabled when cache state is `failed` or `recovering`; Recover Cache disabled when cache state is `healthy`. Matches `setNodeState` guard in SimEngine — no silent no-ops at the UI level.

Split viewport: topology graph (~40% width, left) + 3 metric charts stacked vertically (~60% width, right). Kill/Recover Cache buttons below the graph. User sees node color changes and chart spikes simultaneously — critical for the cascade moment.

**Node circle display:**
- Circle fill color = node state (see color mapping below)
- Inside circle: queue depth percentage (e.g., `80%`) — the number that explains WHY the node is turning amber/red
- Below circle: node name label (LB, App A, App B, Cache, DB)
- LB shows `0%` always (never queues); excluded from charts for same reason

**D3Renderer:** Renders node graph (color-coded circles + SVG edges) + shared 60-second sliding-window time-series panel: one chart per metric (throughput, p99, error rate), one line per node.

**State machine per node:**

| From → To | Trigger |
|-----------|---------|
| healthy → failed | `setNodeState('failed')` (button) — any node |
| failed → recovering | `setNodeState('recovering')` (button) — any node |
| recovering → healthy | Auto after 300 ticks (30s) — any node |
| healthy → degraded | Auto when `queueDepth > 0.8 * capacity` for 5+ consecutive ticks — **DB only (v1)** |
| degraded → failed | Auto when `queueDepth >= capacity` — **DB only (v1)** |

**Constants (all per-node values — tune after first run):**
```ts
const TICK_DURATION = 0.1          // seconds (10 ticks/s)
const CACHE_HIT_RATE_HEALTHY = 0.90
const CACHE_HIT_RATE_FAILED  = 0.00

// Node capacities (req/s ceiling before queue starts filling)
const CAPACITY = { lb: 5000, appA: 2000, appB: 2000, cache: 3000, db: 400 }

// Base latency per node (ms, healthy + empty queue)
const BASE_LAT  = { lb: 1, appA: 5, appB: 5, cache: 2, db: 20 }

// Initial state: all queueDepth = 0
```

**Recovering node behavior:** A recovering node uses the `healthy` throughput formula (`effectiveThroughput = incomingLoad`) immediately — recovery is instant from a throughput standpoint, just gated on the 300-tick timer before state flips back to `healthy`. This means metrics normalize quickly after pressing "Recover Cache," which is intentional for a learning tool (you want to see the recovery happen, not wait for it).

**App server degradation:** In v1, app servers do not independently degrade. Their error rate and latency are derived solely from DB metrics (1-tick lag). App server `queueDepth` is always 0. This simplification is intentional — the v1 lesson is "cache dies → DB overloads → everything breaks," not app server capacity limits.

**LB on charts:** Load Balancer is excluded from the time-series charts. It routes only; its metrics are always flat (it never queues). Charts show 4 lines: App A, App B, Cache, DB.

**Tick execution order — `incomingLoad` routing (run before per-node formulas each tick):**
```ts
// Topological order: lb → appA → appB → cache → db
const cacheMissRate = nodes.cache.state === 'failed' ? 1.0 : (1 - CACHE_HIT_RATE_HEALTHY)
const totalAppThroughput = nodes.appA.effectiveThroughput + nodes.appB.effectiveThroughput

nodes.lb.incomingLoad    = BASE_LOAD
nodes.appA.incomingLoad  = nodes.lb.effectiveThroughput * 0.5
nodes.appB.incomingLoad  = nodes.lb.effectiveThroughput * 0.5
nodes.cache.incomingLoad = totalAppThroughput
nodes.db.incomingLoad    = totalAppThroughput * cacheMissRate
// Note: when cache is 'failed', set queueDelta = 0 (node is dead; don't accumulate queue)
// This keeps cache charts clean so the user's eye goes to DB, not cache.
```

Derivation check (matches cascade walkthrough):
- Healthy: `db.incomingLoad = 1000 × 0.1 = 100 req/s` ✓
- Kill Cache: `cacheMissRate = 1.0 → db.incomingLoad = 1000 req/s` ✓

**Load propagation formulas (per tick, applied after routing above):**
```ts
// Capacity cap is what generates queue pressure — without it, healthy nodes process
// unlimited load and queueDelta is always 0 (cascade never fires).
effectiveThroughput =
  state === 'failed'    ? 0 :
  state === 'degraded'  ? incomingLoad * 0.3 :
  state === 'recovering' ? Math.min(incomingLoad, capacity) :
  /* healthy */            Math.min(incomingLoad, capacity)

// accumulates when incoming > capacity; drains when incoming < capacity (queueDelta goes negative)
// Failed nodes don't accumulate queue — they're dead (chart focus stays on DB)
queueDelta = state === 'failed' ? 0 : (incomingLoad - effectiveThroughput) * TICK_DURATION
queueDepth = Math.max(0, Math.min(capacity, queueDepth + queueDelta))

p99Latency =
  state === 'failed'    ? baseLat :           // node dead — no meaningful latency
  state === 'recovering' ? baseLat :           // same as healthy — intentional
                           baseLat * (1 + queueDepth / capacity)

errorRate =
  state === 'failed'    ? 1.0 :
  state === 'degraded'  ? 0.2 + (queueDepth / capacity) * 0.6 :
  state === 'recovering' ? 0.0 :              // same as healthy — intentional
                           0.0

// App servers inherit DB metrics with 1-tick lag
appServer.errorRate[t]   = db.errorRate[t - 1]
appServer.p99Latency[t]  = appServer.baseLat + db.p99Latency[t - 1]
```

**Cache failure cascade walkthrough:**
- Baseline: 1000 req/s total. LB splits 500/500. Cache hit rate 90% → DB sees ~100 req/s. All green.
- Kill Cache: hit rate → 0%. Both app servers route all reads to DB → DB load jumps to ~1000 req/s.
- DB capacity = 400. Queue fills at (1000−400)×0.1 = 60 req/s per tick.
- After 5 ticks above 80% queue: DB → degraded. App server metrics lag 1 tick behind.
- If queue fills: DB → failed. App servers hit 100% error rate.
- Recover Cache: DB load drops to ~100 req/s. Queue drains at (400−100)×0.1 = 30 req/tick. Full drain in ~10–30s.

**Open question resolved:** Injection is instantaneous; cascade is gradual via tick loop.

## Reviewer Concerns (unresolved after 3 review rounds)

Minor items to tune during implementation — not blockers:
- Capacity values for LB, App Servers, Cache are educated guesses. Adjust until the failure cascade feels realistic.
- `baseLat` values are approximations. DB at 20ms healthy is reasonable; tune to taste.
- The "recovering" throughput choice (same as healthy, immediate) is a simplification. If the recovery feels too snappy, add a linear ramp from 30% to 100% over the 300-tick window.

## Open Questions

1. What baseline load numbers feel realistic? 1000 req/s is a teaching abstraction — the exact number matters less than the ratios (cache hit rate, DB capacity ceiling). Tune after first run.
2. ~~Should node failure be instantaneous or gradual?~~ **Resolved:** Injection is instantaneous; cascade is gradual via tick loop.
3. ~~What's the right tick rate?~~ **Resolved:** 10 ticks/second.

**Viewport:** Designed for 1024px+ desktop. Mobile is not supported in v1. Add a notice below 768px viewport width: *"Open on a desktop browser for the full simulation."*

**Accessibility (v1 minimum):**
- Buttons: `aria-label="Kill Cache"` / `aria-label="Recover Cache"` + `aria-disabled="true"` when disabled
